## 1. 当前推荐流程顺序（含依赖）

文档《开发计划》**阶段总览**的叙述顺序为：P0 → P1 → P2 → …；但**跨阶段依赖图**标明 **P2 → P1**（标准图库先于设计深度联动）。  
**监管用实施顺序**建议：P0 → **P2 与 P1 可并行启动**，但 **M2 须先于 M1 中含「标准件调用 / 选型与图库联动」的联调验收**，以避免返工。

- **P0** 为唯一起点，完成后才能全面开展 P1、P2。
- **P2 → P1（能力依赖）**：标准图库未稳定时，设计侧深度选型易返工。
- **P1 → P4、P1 → P5；P2 → P3；P3 → P5；P4 → P5**：工艺现场与研发对接在 P5 前需有可用主线，**P5 集成与看板**为汇聚阶段。

**监管用一句话顺序**：  
`G0 启动定案` → `P0 + 门禁M0` → `P2 ∥ P1（先保 M2，再收口 M1 中与图库联动部分）` → `联合门禁 M1+M2` → `P3` → `P4` → `P5 + 门禁M5` → `P6 持续`；**每 2 周迭代末演示**（见《开发计划》§7）。  
**进度注记（2026-04-13）**：P1/P2/P3/P4/P5 **开发向 MVP 骨架已在仓库落地**（含 P4 **`/field`**；P5 **驾驶舱 KPI + 集成作业桩** **`/dashboard/summary`**、**`/integration`**）；真实 ERP 联调、企业级审批引擎、监管 **M1～M5 正式签字**仍按清单分项收口。

## 2. 监管门禁（Gate）定义

| 门禁 | 触发时机 | 监管要点（准出条件示例） |
|------|------------|--------------------------|
| **G0** | 写代码前（或 P0 后追溯） | 技术栈（React+Tailwind、Python）与仓库形态已定；**MVP 范围**书面确认；**ERP 物料/BOM 调研清单**已输出；dev 库、对象存储、CI 可跑。**若 P0 先行**：允许「文书追溯」，未决项须在 P1 铺开前责任到人（见 §3 G0）。 |
| **M0** | P0 结束 | 可演示后台框架；组织/岗位/角色、登录、审计、文件上传、通知骨架可用；**业务 UAT 签字或纪要** |
| **M1** | P1 结束 | 「项目 → 设计任务 → 选型存根」端到端可演示；变更流程至少跑通简化版（**当前代码：状态字段可演示，无审批引擎**） |
| **M2** | P2 结束 | 标准件发布态可被设计模块引用；版本不可原地篡改（新版本迭代）；入库/淘汰审批接通（**当前代码：主数据+图纸版本 CRUD 可演示；引用与审批流未接通**） |
| **M3** | P3 结束 | 成果入库申请进入标准库审批闭环（**当前代码**：入库审批 API + 通过后写入标准件；**正式签字**仍以 §3 P3「监管 M3」为准） |
| **M4** | P4 结束 | 试模/售后工单状态机从发起到关闭可查询；溯源关联可用（**当前代码**：`/field` 下试模/售后 **状态迁移校验** + 列表筛选；**报告模板/打印**与**正式签字**仍为业务侧待办） |
| **M5** | P5 结束 | ERP（及可选 MES）试点联调通过；驾驶舱/KPI 可看数（**当前代码**：KPI 聚合 + 集成作业**桩**可演示；**真实对接**仍以试点纪要为准） |
| **G-迭代** | 每 2 周末 | 演示、风险登记、范围变更走变更记录；发布检查迁移脚本与回滚预案 |
